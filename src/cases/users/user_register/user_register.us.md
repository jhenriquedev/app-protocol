# US: user_register

## Capability

As a caller, I want to register a new user through the canonical flow so I can validate input, persist the account, and trigger post-success processing consistently.

## Context

- This Case is composed.
- It belongs to the `users` domain.
- It depends on `user_validate` through `ctx.cases`.
- The canonical execution path is `user_register.api`.

## Input Requirements

- `email`: required string
- `name`: required string
- `password`: required string

## Output Requirements

The Case returns the created user with:

- `id`
- `email`
- `name`
- `createdAt`

Expected output rules:

- `id` is generated by the backend
- `createdAt` is generated at persistence time
- password is never returned

## Validation Rules

- `email` is required
- `name` is required
- `password` is required
- `password` must have at least `8` characters
- registration must fail when `user_validate` returns semantic errors

## Business Invariants

- registration composes canonical validation before persistence
- registration is mutating
- successful registration emits follow-up processing through `user_register.stream`
- agentic execution requires explicit confirmation

## Surfaces Involved

- `user_register.domain.case.ts`
  - defines schemas, invariants, and examples
- `user_register.api.case.ts`
  - exposes `POST /users/register`
  - composes `user_validate.api` via `ctx.cases`
- `user_register.ui.case.ts`
  - drives the registration form
  - checks password confirmation before calling the backend
- `user_register.stream.case.ts`
  - reacts to `user_registered`
  - represents post-success processing
- `user_register.agentic.case.ts`
  - exposes the Case as a confirmable tool for `apps/agent`
  - delegates execution to `user_register.api`

## Composition

- `user_register.api` composes `users.user_validate.api`
- no direct Case-to-Case import is allowed

## Integrations

### Host integrations

- `apps/backend`
  - route: `POST /users/register`
- `apps/portal`
  - entry point: registration form submit
- `apps/agent`
  - publishes the same capability through HTTP and MCP

## Events / recovery / policy

- post-success host flow produces the `user_registered` event
- `user_register.stream` consumes that event for follow-up processing
- `user_register.agentic` is exposed as tool `user_register`
- MCP publication is enabled
- execution mode: `manual-approval`
- confirmation is required

## Validation Scenarios

### Domain

1. Given valid email, name, and password, the domain accepts the input.
2. Given invalid email, short password, or short name, the domain rejects the input.

### API

1. Given valid input, the API composes validation, persists the user, and returns the created payload.
2. Given validation failure from `user_validate`, the API returns a composition failure with canonical error details.
3. Given invalid required fields, the API returns a structured validation failure.

### UI

1. Given matching passwords and valid fields, the UI submits the registration request and renders success feedback.
2. Given mismatched passwords, the UI blocks submission locally and surfaces the mismatch.

### Stream

1. Given a successful registration event, the stream surface consumes the payload and runs post-success processing.
2. The declared recovery policy remains deterministic and valid for retry and dead-letter handling.

### Agentic

1. The agentic surface exposes discovery, context, prompt, tool, MCP, RAG, and policy metadata.
2. Tool execution delegates to `user_register.api` through `ctx.cases`.
3. The host rejects execution without explicit confirmation.
