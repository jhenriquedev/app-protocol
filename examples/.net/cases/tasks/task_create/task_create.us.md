# US: task_create

## Capability

As a user, I want to create a new task card so I can add work to the board.

## Context

- This Case is atomic.
- It belongs to the `tasks` domain.
- It is triggered from the portal create-task modal.
- The backend persists the new task through `packages/data`.
- The created card appears in the `todo` column.
- This .NET implementation must preserve the same semantics already validated in `examples/react/`.

## Input Requirements

- `title`: required string
- `description`: optional string

Normalization rules:

- `title` must be trimmed before validation
- `description`, when present, may be trimmed before persistence

## Output Requirements

The Case returns the created `Task` with:

- `id`
- `title`
- `description?`
- `status`
- `createdAt`
- `updatedAt`

Expected output rules:

- `status` is always `todo`
- `createdAt` and `updatedAt` are generated by the backend
- `createdAt` and `updatedAt` are equal on first creation

## Validation Rules

- `title` is required after trim
- `title` cannot be an empty string
- `description` is optional
- input must not set `id`
- input must not set `status`
- input must not set timestamps

## Business Invariants

- every created task starts in `todo`
- the backend is the source of truth for identifiers and timestamps
- creating a task must append one new record to the local store
- task creation does not compose with other Cases in v1

## Surfaces Involved

- `task_create.domain.case.cs`
- `task_create.api.case.cs`
- `task_create.ui.case.cs`
- `task_create.agentic.case.cs`

## Composition

- no cross-case composition in v1
- after success, board refresh is handled by the portal flow, not by direct UI-to-UI Case composition

## Integrations

### Host integrations

- `apps/backend`
  - route: `POST /tasks`
  - context: `ApiContext`
- `apps/portal`
  - entry point: create-task modal submit
  - context: `UiContext`
- `apps/agent`
  - tool: `task_create`
  - canonical execution: `task_create.api`

### Package integrations

- `packages/data`
  - reads current task collection
  - appends the new task
  - writes the updated collection to `tasks.json`
  - serializes writes safely when backend and agent share the same store
- `packages/design_system`
  - `CreateTaskButton`
  - `TaskFormModal`

## Events / recovery / policy

- no `stream` surface in v1
- no retries or dead-letter policy in v1
- `task_create.agentic` is exposed as tool `task_create`
- the same tool is published through HTTP, MCP stdio, and remote MCP HTTP by `apps/agent`
- agentic execution mode: `direct-execution`
- confirmation is not required for this mutation in v1

## Validation Scenarios

1. Given a valid title and optional description, the domain accepts the input.
2. Given a blank title such as `"   "`, the domain rejects the input.
3. Given valid input, `POST /tasks` returns success and persists the created task.
4. Given invalid input, the API returns a structured validation error.
5. Given successful creation, the portal closes the modal and refreshes the board.
6. Tool execution delegates to `task_create.api` through `ctx.cases`.
7. MCP `tools/call` preserves the same structured validation failures as the HTTP agent host.
