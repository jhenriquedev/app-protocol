# US: task_create

## Capability

As a user, I want to create a new task card so I can add work to the board.

## Context

- This Case is atomic.
- It belongs to the `tasks` domain.
- It is triggered from the portal create-task modal.
- The backend persists the new task through `packages/data`.
- The created card appears in the `todo` column.

## Input Requirements

- `title`: required string
- `description`: optional string

Normalization rules:

- `title` must be trimmed before validation
- `description`, when present, may be trimmed before persistence

## Output Requirements

The Case returns the created `Task` with:

- `id`
- `title`
- `description?`
- `status`
- `createdAt`
- `updatedAt`

Expected output rules:

- `status` is always `todo`
- `createdAt` and `updatedAt` are generated by the backend
- `createdAt` and `updatedAt` are equal on first creation

## Validation Rules

- `title` is required after trim
- `title` cannot be an empty string
- `description` is optional
- input must not set `id`
- input must not set `status`
- input must not set timestamps

## Business Invariants

- every created task starts in `todo`
- the backend is the source of truth for identifiers and timestamps
- creating a task must append one new record to the local store
- task creation does not compose with other Cases in v1

## Surfaces Involved

- `task_create.domain.case.ts`
  - defines the semantic input and output contracts
  - defines validation for required title
- `task_create.api.case.ts`
  - exposes `POST /tasks`
  - delegates persistence to the local data package
- `task_create.ui.case.ts`
  - drives the create-task modal submit action
  - sends the creation request to the backend API
- `task_create.agentic.case.ts`
  - exposes task creation as a discoverable tool for `apps/agent`
  - delegates execution to `task_create.api`

## Composition

- no cross-case composition in v1
- after success, board refresh is handled by the portal flow, not by direct UI-to-UI Case composition

## Integrations

### Host integrations

- `apps/backend`
  - route: `POST /tasks`
  - context: `ApiContext`
- `apps/portal`
  - entry point: create-task modal submit
  - context: `UiContext`

### Package integrations

- `packages/data`
  - reads current task collection
  - appends the new task
  - writes the updated collection to `tasks.json`
  - serializes writes safely when `apps/backend` and `apps/agent` share the same store
- `packages/design_system`
  - `CreateTaskButton`
  - `TaskFormModal`

## Events / recovery / policy

- no `stream` surface in v1
- no retries or dead-letter policy in v1
- `task_create.agentic` is exposed as tool `task_create`
- the same tool is published through HTTP and MCP by `apps/agent`
- agentic execution mode: `direct-execution`
- confirmation is not required for this mutation in v1

## Validation Scenarios

### Domain

1. Given a valid title and optional description, the domain accepts the input.
2. Given a blank title such as `"   "`, the domain rejects the input.
3. Given client-supplied `status` or timestamp fields, the domain rejects the input contract.

### API

1. Given `POST /tasks` with valid input, the API returns success and the created task with status `todo`.
2. Given valid input, the API persists the created task to `packages/data/tasks.json`.
3. Given invalid input with blank title, the API returns a structured validation error.
4. Given consecutive create requests, the API appends both tasks without overwriting previous records.

### UI

1. Given the create-task modal is open, submitting valid data triggers the backend request.
2. Given successful creation, the modal closes and the board refresh flow can render the new card in `todo`.
3. Given invalid input, the UI keeps the user in the create flow and surfaces the failure state.
4. Given repeated submit actions while a request is already in flight, the UI dispatches at most one create request.

### Agentic

1. The agentic surface exposes discovery, context, prompt, tool, MCP, and policy metadata.
2. Tool execution delegates to `task_create.api` through `ctx.cases`.
3. The tool returns the created task payload rather than a shadow response shape.
4. Structured API validation failures propagate as `AppCaseError` instead of degrading to generic agent runtime errors.
5. MCP `tools/call` preserves the same structured validation failure contract as the HTTP agent host.

### Integration

1. Portal create action -> backend `POST /tasks` -> `packages/data` write -> board reload shows the new card.
2. Restarting the backend after a successful create preserves the card because persistence is local and durable.
3. Concurrent create requests from `apps/backend` and `apps/agent` preserve every task without corrupting `tasks.json`.
