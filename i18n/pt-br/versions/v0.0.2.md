# AI-First Programming Protocol (APP)

## Specification v0.0.2

---

# 1. Overview

**AI-First Programming Protocol (APP)** é um protocolo arquitetural para desenvolvimento de software otimizado para **colaboração entre humanos e agentes de IA**.

O APP organiza sistemas em unidades autocontidas chamadas **Cases**, cada uma representando uma capacidade específica do sistema.

Cada Case pode possuir múltiplas **surfaces**, que definem como essa capacidade é expressa e executada:

* domínio semântico
* execução backend
* execução de interface
* execução orientada a eventos
* operação agentic

A versão **v0.0.2** introduz o **Agentic Protocol**, que torna Cases **descobertos, compreendidos e executáveis por agentes de IA**, sem quebrar o princípio de unidade autocontida.

---

# 2. Objectives

APP foi criado para resolver problemas comuns em sistemas desenvolvidos com suporte de IA:

* alto custo de contexto
* estrutura imprevisível
* navegação excessiva
* lógica dispersa entre camadas
* dificuldade de agentes compreenderem código

APP prioriza:

* **unidades autocontidas**
* **estrutura previsível**
* **baixo custo de contexto**
* **semântica clara**
* **interoperabilidade com IA**

---

# 3. Core Principles

## 3.1 Case as the Canonical Unit

A unidade fundamental do APP é o **Case**.

Um Case representa uma **capacidade autocontida do sistema**.

Exemplos:

* `user_validate`
* `get_user_by_id`
* `invoice_pay`
* `theme_toggle`

Cada Case possui sua própria pasta e pode conter múltiplas surfaces.

---

## 3.2 Self-Contained Architecture

Um Case deve conter tudo o que é necessário para implementar sua capacidade, exceto dependências compartilhadas do `core`.

Isso permite:

* fácil compreensão
* baixo custo de contexto
* geração de código por agentes
* isolamento arquitetural

---

## 3.3 Surface-Oriented Structure

Cada Case pode possuir múltiplas **surfaces**, que representam diferentes formas de execução da mesma capacidade.

Surfaces definidas pelo protocolo:

* `domain.case.ts`
* `api.case.ts`
* `ui.case.ts`
* `stream.case.ts`
* `agentic.case.ts`

---

## 3.4 Predictable Structure

Todos os Cases devem seguir uma estrutura consistente para facilitar:

* leitura humana
* geração automática
* raciocínio por IA

---

# 4. Canonical Project Structure

Estrutura recomendada:

```text
app/
├── App.ts
├── core/
│   ├── domain.case.ts
│   ├── api.case.ts
│   ├── ui.case.ts
│   ├── stream.case.ts
│   ├── agentic.case.ts
│   └── shared/
│       ├── http.ts
│       ├── logger.ts
│       ├── result.ts
│       ├── storage.ts
│       ├── test.ts
│       └── utils.ts
└── cases/
    ├── cases.ts
    └── users/
        └── user_validate/
            ├── user_validate.domain.case.ts
            ├── user_validate.api.case.ts
            ├── user_validate.ui.case.ts
            ├── user_validate.stream.case.ts
            └── user_validate.agentic.case.ts
```

---

# 5. Case Structure

Cada Case deve possuir uma pasta própria:

```text
cases/
  users/
    user_validate/
```

Arquivos dentro do Case seguem o padrão:

```
<case_name>.<surface>.case.ts
```

Exemplo:

```
user_validate.api.case.ts
```

---

# 6. Case Surfaces

## 6.1 Domain Surface

Arquivo:

```
<case>.domain.case.ts
```

Contém a **semântica pura do Case**.

Responsabilidades:

* value objects
* enums de domínio
* regras puras
* invariantes
* validações
* estruturas semânticas

Proibido:

* acesso a banco
* HTTP
* logging
* side effects

---

## 6.2 API Surface

Arquivo:

```
<case>.api.case.ts
```

Representa execução backend síncrona.

Responsabilidades:

* parsing de input
* autorização
* orquestração
* execução da capacidade
* transformação de resposta

Estrutura sugerida:

```
handler
router
test
```

Seções internas comuns:

```
_validate
_authorize
_service
_repository
_present
```

---

## 6.3 UI Surface

Arquivo:

```
<case>.ui.case.ts
```

Representa interface reativa.

Responsabilidades:

* estado local
* interação do usuário
* renderização
* efeitos
* integração com APIs

Estrutura sugerida:

```
render
mount
test
```

---

## 6.4 Stream Surface

Arquivo:

```
<case>.stream.case.ts
```

Representa execução orientada a eventos.

Responsabilidades:

* consumo de eventos
* publicação
* pipelines
* retries
* idempotência

Estrutura sugerida:

```
handler
subscribe
test
```

---

# 7. Agentic Protocol (New in v0.0.2)

A v0.0.2 introduz a **surface agentic**.

Arquivo:

```
<case>.agentic.case.ts
```

Essa surface define **como o Case é compreendido e operado por agentes de IA**.

Ela não substitui as surfaces tradicionais.
Ela descreve e expõe a capacidade do Case para sistemas agentic.

---

# 8. Agentic Surface Responsibilities

A surface agentic define:

* descoberta semântica
* contexto de execução
* prompt estruturado
* exposição como tool
* integração com MCP
* regras de recuperação contextual (RAG)
* políticas de uso

---

# 9. Agentic Protocol Structure

Estrutura conceitual:

```typescript
export const userValidateAgenticCase = {

  discovery: {},

  context: {},

  prompt: {},

  tool: {},

  mcp: {},

  rag: {},

  policy: {},

  examples: []

}
```

Cada seção define um aspecto da interação agentic.

---

# 10. Discovery

Define como agentes encontram e identificam o Case.

Exemplo:

```
name
description
tags
capabilities
aliases
category
```

Objetivo:

* permitir descoberta automática
* facilitar classificação semântica

---

# 11. Context

Define o contexto mínimo necessário para executar o Case corretamente.

Pode incluir:

* autenticação requerida
* escopo de tenant
* dependências
* restrições
* pré-condições

---

# 12. Prompt

Define instruções estruturadas para agentes utilizarem o Case.

Conteúdo típico:

* objetivo
* quando usar
* quando não usar
* formato esperado
* limitações

Prompts no APP são **artefatos estruturados**, não texto arbitrário.

---

# 13. Tool

Define o contrato de execução do Case como ferramenta para IA.

Inclui:

* nome canônico
* descrição
* schema de input
* schema de output
* mapeamento para execução real

Regra fundamental:

> A Tool deve sempre executar a implementação canônica do Case.

---

# 14. MCP

Define como o Case é exposto via **Model Context Protocol**.

MCP é um adaptador de interoperabilidade para runtimes agentic.

Responsabilidades:

* registrar tool
* mapear contexto
* aplicar políticas
* traduzir contrato do Case

---

# 15. RAG

Define informações de recuperação contextual.

Pode incluir:

* fontes de conhecimento relevantes
* documentos permitidos
* hints de recuperação
* escopo semântico

Objetivo:

* melhorar a compreensão do Case por agentes
* fornecer contexto externo quando necessário

---

# 16. Policy

Define restrições e guardrails.

Exemplos:

* confirmação obrigatória
* limites de execução
* restrições de segurança
* escopo permitido

---

# 17. Examples

Define exemplos determinísticos de uso.

Exemplos ajudam agentes a:

* entender comportamento esperado
* reduzir ambiguidade
* melhorar raciocínio contextual

---

# 18. Case Dependencies

Um Case pode depender apenas de:

* `core`
* `core/shared`
* seu próprio `domain.case.ts`

Casos especiais:

`agentic.case.ts` pode também referenciar:

* `api.case.ts`
* `stream.case.ts`

como ponto de execução real.

---

# 19. Case Registry

Arquivo:

```
cases/cases.ts
```

Responsável por registrar todos os Cases do sistema.

Pode ser usado para:

* bootstrap
* descoberta
* roteamento
* geração de documentação

---

# 20. AI Operation Model

No APP v0.0.2, agentes devem operar Cases seguindo o fluxo:

1. descobrir capacidade
2. compreender contexto
3. interpretar prompt
4. validar políticas
5. executar tool
6. retornar resultado

---

# 21. AI-First Development Flow

Fluxo de criação de novas capacidades:

1. definir intenção
2. criar Case folder
3. implementar domínio
4. implementar superfícies necessárias
5. adicionar surface agentic
6. registrar Case

---

# 22. Anti-Patterns

Arquiteturas que violam o protocolo:

### Layer-First Architecture

```
api/
domain/
services/
```

### God Case

Um único arquivo contendo tudo.

### Shadow Tools

Tools com lógica diferente da execução canônica.

### Hidden Prompts

Prompts espalhados em strings arbitrárias.

# 24. Canonical Definition

> No AI-First Programming Protocol (APP), **Case** é a unidade canônica de organização do software. Cada Case representa uma capacidade autocontida do sistema e pode conter múltiplas surfaces que definem sua semântica, execução e operação agentic, permitindo que humanos e agentes de IA compreendam e utilizem o sistema com baixo custo de contexto.
