# AI-First Programming Protocol (APP)

## Specification v0.0.1

---

# 1. Overview

**AI-First Programming Protocol (APP)** é um protocolo arquitetural para organização de software otimizado para **desenvolvimento assistido por IA e pair programming com agentes**.

O APP define uma forma previsível de estruturar código baseada em **Cases**, unidades autocontidas que representam capacidades específicas de um sistema.

Cada Case encapsula:

* semântica de domínio
* superfícies de execução
* organização estrutural previsível
* dependência mínima de contexto

O objetivo do APP é permitir que humanos e agentes de IA possam:

* entender rapidamente qualquer parte do sistema
* modificar código com baixo custo de contexto
* gerar novas capacidades de forma consistente
* evitar navegação excessiva entre arquivos
* manter isolamento semântico entre capacidades

---

# 2. Motivation

Codebases modernas apresentam problemas estruturais que dificultam desenvolvimento assistido por IA:

* dispersão de lógica entre múltiplas camadas
* dependências implícitas difíceis de inferir
* estruturas inconsistentes entre módulos
* excesso de boilerplate
* custo elevado de navegação entre arquivos

APP resolve esses problemas introduzindo uma unidade estrutural previsível chamada **Case**.

---

# 3. Core Concepts

## 3.1 Case

**Case** é a unidade canônica do APP.

Um Case representa uma **capacidade autocontida do sistema**.

Exemplos de Cases:

* `user_validate`
* `get_user_by_id`
* `invoice_pay`
* `theme_toggle`

Cada Case:

* vive em uma pasta própria
* pode conter múltiplas superfícies
* possui dependência mínima
* pode conter um núcleo semântico opcional

---

## 3.2 Case Folder

Todo Case deve existir dentro de uma **Case Folder dedicada**.

Mesmo que exista apenas um arquivo.

Exemplo:

```text
cases/
  users/
    user_validate/
      user_validate.api.case.ts
```

A pasta garante:

* previsibilidade estrutural
* isolamento semântico
* escalabilidade da arquitetura
* melhor indexação por agentes

---

## 3.3 Case Surfaces

Um Case pode possuir **superfícies de execução**, que representam formas de interação com a capacidade.

As superfícies canônicas do APP são:

* `api`
* `ui`
* `stream`

---

## 3.4 Case Domain

Um Case pode possuir um núcleo semântico opcional chamado **Domain**.

Arquivo:

```
domain.case.ts
```

O Domain representa a semântica pura da capacidade.

---

# 4. Canonical Project Structure

Estrutura recomendada de projeto:

```text
app/
├── App.ts
├── core/
│   ├── api.case.ts

│   └── shared/

│       ├── result.ts
│       ├── storage.ts
│       ├── test.ts
│       └── utils.ts

    ├── cases.ts

            ├── user_validate.domain.case.ts
            ├── user_validate.api.case.ts
            ├── user_validate.ui.case.ts
            └── user_validate.stream.case.ts
```

# 5. Case Types

APP v0.0.1 define quatro partes canônicas de um Case.

## 5.1 Domain Case

Arquivo:

```
<case>.domain.case.ts
```

### Responsabilidade

Contém a **semântica pura da capacidade**.

Pode incluir:

* value objects
* enums de domínio
* regras puras de validação
* estruturas semânticas
* invariantes
* transformações puras
* serialização estrutural
* copyWith
* igualdade semântica

### Proibido

Domain não pode conter:

* acesso a banco
* acesso HTTP
* logger
* filesystem
* event bus
* renderização
* side effects
* handlers

### Exemplo

```ts
export enum UserValidationStatus {
  Valid = "valid",
  Invalid = "invalid"
}

export class UserDocument {
  constructor(readonly value: string) {}

  isValid(): boolean {
    return this.value.length > 0
  }
}
```

---

# 5.2 API Case

Arquivo:

```
<case>.api.case.ts
```

Representa a **superfície síncrona de backend**.

Exemplos:

* REST endpoints
* RPC handlers
* server functions

### Responsabilidades

* parse de input
* autorização
* orquestração
* chamada de serviços
* transformação de resposta
* acesso a infraestrutura

### Estrutura sugerida

Métodos públicos:

```
handler
router
test
```

Seções internas sugeridas:

```
_validate
_authorize
_service
_repository
_present
```

---

# 5.3 UI Case

Arquivo:

```
<case>.ui.case.ts
```

Representa a **superfície de interface reativa**.

### Responsabilidades

* estado local
* ações do usuário
* efeitos
* renderização
* integração com API

### Estrutura sugerida

Métodos públicos:

```
render
mount
test
```

Seções internas sugeridas:

```
_state
_actions
_effects
_view
_queries
```

---

# 5.4 Stream Case

Arquivo:

```
<case>.stream.case.ts
```

Representa execução orientada a eventos.

### Exemplos

* eventos
* filas
* webhooks
* realtime
* subscriptions
* pipelines

### Responsabilidades

* consumo de eventos
* transformação
* publicação
* retry
* idempotência

### Estrutura sugerida

```
handler
subscribe
test
```

Seções internas:

```
_consume
_transform
_publish
_retry
_policy
```

---

# 6. Case Dependencies

Um Case deve ter dependências restritas.

### Dependências permitidas

* `core`
* `core/shared`
* `domain.case.ts` do próprio Case
* dependências externas autorizadas pelo runtime

### Dependências proibidas

Um Case **não deve depender de outro Case diretamente**.

Se duas capacidades compartilham semântica, ela deve ser promovida para:

```
core/shared
```

ou

```
domain.case.ts
```

---

# 7. Core Layer

A pasta `core` contém as bases do protocolo.

Arquivos:

```
core/
  api.case.ts
  ui.case.ts
  stream.case.ts
  domain.case.ts
```

Esses arquivos definem:

* classes base
* interfaces base
* contratos de contexto
* abstrações do protocolo

Eles **não contêm lógica de negócio de Cases**.

---

# 8. Shared Core

```
core/shared
```

Contém utilidades compartilhadas.

Exemplos:

* http client
* logger
* result helpers
* storage
* testing helpers
* utilities

Regra:

Se algo não pertence semanticamente a um Case específico, pode ir para `core/shared`.

---

# 9. Case Registry

Arquivo:

```
cases/cases.ts
```

Responsável por registrar Cases.

Pode ser usado para:

* bootstrap
* binding de rotas
* binding de eventos
* geração de documentação
* descoberta de capacidades
* indexação para IA

Exemplo conceitual:

```ts
export const cases = {
  users: {
    user_validate: {
      api: () => import("./users/user_validate/user_validate.api.case"),
      ui: () => import("./users/user_validate/user_validate.ui.case"),
      stream: () => import("./users/user_validate/user_validate.stream.case"),
      domain: () => import("./users/user_validate/user_validate.domain.case")
    }
  }
}
```

---

# 10. Naming Convention

Formato canônico:

```
<case_name>.<surface>.case.ts
```

Exemplos:

```
user_validate.api.case.ts
user_validate.ui.case.ts
user_validate.stream.case.ts
user_validate.domain.case.ts
```

A pasta deve usar o mesmo nome do Case:

```
user_validate/
```

---

# 11. Minimal Case Rule

Nem todo Case precisa de todas as superfícies.

Exemplos válidos.

### Apenas UI

```
theme_toggle/
  theme_toggle.ui.case.ts
```

### Domain + API

```
user_validate/
  user_validate.domain.case.ts
  user_validate.api.case.ts
```

### Domain + Stream

```
invoice_paid/
  invoice_paid.domain.case.ts
  invoice_paid.stream.case.ts
```

### Full Stack

```
user_validate/
  user_validate.domain.case.ts
  user_validate.api.case.ts
  user_validate.ui.case.ts
  user_validate.stream.case.ts
```

---

# 12. Infrastructure Injection

Infraestrutura não deve ser criada diretamente dentro do Case.

Ela deve ser fornecida por contexto.

Exemplos de capacidades injetadas:

```
db
http
logger
cache
config
auth
eventBus
storage
```

Essa regra reduz:

* boilerplate
* divergência estrutural
* custo de contexto para IA

---

# 13. Anti-Patterns

## Layer-first filesystem

Errado:

```
api/
domain/
ui/
stream/
```

---

## God Case

Um único arquivo contendo:

* domínio
* API
* UI
* streams
* infraestrutura

---

## Fake Domain

Um `domain.case.ts` contendo:

* DTO HTTP
* acesso a banco
* event envelopes
* adapters

---

## Generic Buckets

Arquivos como:

```
helpers.ts
types.ts
models.ts
contracts.ts
```

sem dono semântico.

---

# 14. Framework Independence

APP é independente de framework.

Pode ser usado com:

* Node backends
* serverless
* React
* Vue
* Svelte
* Solid
* realtime systems
* full-stack frameworks

O framework é apenas o host de execução.

---

# 15. AI Interaction Model

Um agente compatível com APP deve assumir:

* cada Case é uma unidade autocontida
* `domain.case.ts` contém semântica pura
* `api.case.ts` contém execução backend
* `ui.case.ts` contém interface
* `stream.case.ts` contém eventos
* `core` contém infraestrutura do protocolo
* `cases.ts` contém descoberta do sistema

Isso permite que agentes trabalhem com **baixo custo de contexto**.

---

# 16. Canonical Definition

> No AI-First Programming Protocol (APP), **Case** é a unidade canônica de organização do software.
> Cada Case representa uma capacidade autocontida do sistema e pode conter superfícies protocoladas como `domain`, `api`, `ui` e `stream`, mantendo dependência apenas de `core`, `core/shared` e de seu próprio domínio local.

---

# 17. Version Status

**APP v0.0.1** estabelece:

* conceito de Case
* estrutura canônica de pastas
* superfícies do Case
* regras de dependência
* estrutura mínima do protocolo

Ainda não define:

* DSL oficial do registry
* tooling de lint do protocolo
* geração automática de Cases
* manifest formal
* runtime padrão

Esses pontos ficam para versões futuras.
